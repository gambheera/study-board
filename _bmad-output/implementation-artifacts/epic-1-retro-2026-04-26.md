## What Could Have Gone Better

### 1. No dependency version audit before starting
We discovered Riverpod 4.x naming changes and google_sign_in v7 breaking API mid-implementation, causing rework across multiple stories. A pre-epic library spike would have caught these.

### 2. Deferred bug accumulation
8 bugs deferred out of Story 1.6 (DEF1–DEF8) meant Stories 1.7 and 1.8 were built on an unstable auth foundation. We resolved them, but the archaeology was costly.

### 3. Story 1.7 / 1.8 boundary collapse
LoginScreen, `signInWithEmailPassword`, and session restore all landed in Story 1.7's diff. We absorbed the scope creep silently instead of flagging it, making code review harder and story tracking misleading.

### 4. Recurring sync queue gaps
Stories 1.3 and 1.5 had sync queue omissions (TaskDao raw string literals, SurveyDao/QuizDao missing enqueue, updateFcmToken missing enqueue). Each caught in review — but a checklist would have caught them earlier.

### 5. Mounted guards as a pattern, not a review comment
`setState` mounted guards were flagged in Stories 1.6 and 1.8 in review. This should have been a known pattern from Story 1.1 onward.

### 6. No product demo loop
The Project Lead had no visibility into what was built story by story. Implementation files and test results are not a substitute for seeing the actual product.

## Key Lessons Learned — Taking Forward

### Technical

| Lesson | Context |
|---|---|
| `GoogleSignInService` interface pattern — wrap third-party singletons for testability | Story 1.7 — google_sign_in v7 singleton couldn't be injected directly |
| Dart closure type promotion — `final resolved = nullable;` before closures | Story 1.7 — `idToken` type error inside `trySupabase` closure |
| Router redirect with `isLoading` guard — `if (authValue.isLoading) return null` | Story 1.8 — prevents premature redirects on cold start |
| Single `transaction()` for all multi-table Drift writes | Story 1.9 review — `updateProfileWithSubjects()` collapsed from two transactions to one |
| `_explicitSignOut` flag pattern — set flag before action when stream can't distinguish intent | Story 1.8 — distinguish explicit signOut from expired session in `onAuthStateChange` stream |
| `dart run build_runner build --delete-conflicting-outputs` from `studyboard_mobile/` | Correct command and directory — not `flutter pub run`, not from root |

### Process

| Lesson | Context |
|---|---|
| Dependency version audit before every epic | Riverpod 4.x + google_sign_in v7 caught mid-implementation |
| Story boundary enforcement — flag scope creep, don't absorb | Stories 1.7/1.8 code mixing |
| Sync queue checklist for every Drift write on user-owned data | Repeated omissions in 1.3, 1.5 |
| Cap deferred bugs — resolve before building on unstable foundation | DEF1–DEF8 accumulated from 1.6 |
| Test alongside delivery — not after | Test count grew 5× by treating tests as part of done |
| Interface-first for any new third-party service | GoogleSignInService pattern |

## Supabase Assessment

**Strengths:**
- RLS policies with `auth.uid()` — elegant, enforced at the right layer
- Auto JWT refresh — zero ceremony, worked transparently
- `onAuthStateChange` stream — powerful real-time auth state

**Risks to monitor:**
- `on_auth_user_created` trigger creates `public.students` row — client cannot verify trigger success; silent trigger failure would cause user-visible inconsistency
- `onAuthStateChange` `signedIn` event fires before Drift row exists — requires `getCurrentUser()` confirmation before emitting `authenticated` state (DEF8 pattern)
- Mocking Supabase with mocktail requires careful setup — `registerFallbackValue` for enums, wrapper interfaces for `GoTrueClient`
